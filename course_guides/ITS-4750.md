# ITS 4750 Labs

ITS 4750 Internet Engineering is a hands-on, lab-intensive course covering the fundamentals of how the large scale Internet networking is architected and built. You will configure real production-grade software (VyOS routers, pfSense firewalls, Linux servers) in a simulated environment that mirrors ISP and enterprise networking practices and use those environments to explore network architectures. 

# Preface and Warnings

## Variable replacement

Through ALL ITS labs and as a general rule in public documentation various forms of brackets `[ ] < > { }" and quotes are used to signify a variable that must be replaced by the user.  The student is responsible for understanding the context of these strings, but unless otherwise specified the brackets are removed during the replacement.  For example:

`{Ohio-University-Mascot}` becomes `bobcat` NOT `{bobcat}`  

## Defining requirement levels

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED",  "MAY", and "OPTIONAL" in this course are to be interpreted as described in [RFC2119](https://datatracker.ietf.org/doc/rfc2119/).

# Task 3 - Virtual lab environment

GNS3 (Graphical Network Simulator-3) is an open-source network emulation platform that runs real network operating system images - such as VyOS and pfSense - inside a virtualized environment. It lets you build, configure, and test multi-device network topologies without physical hardware. In ITS 4750 it is the primary lab platform, where you will construct and troubleshoot enterprise networks.

It is NOT RECOMMENDED to install GNS3 on a personal computer.  The ITS department's verions of GNS3 has been customized to serve the departments configurations and has been pre-loaded with key machine templates that are not imediatly available on home installations.

- [GNS3 - The software that empowers network professionals](https://www.gns3.com/)
- [Getting Started with GNS3](https://docs.gns3.com/docs/)

## Accessing the ITS GNS3

Because GNS3 projects can be resource-intensive, the university provides each student their own dedicated virtual machine referred to as a gHost.  The gHost is a Linux Ubuntu Desktop Virtual Machine hosted in the ITS datacenter. The student accesses the gHost remotely, either through a web browser via Guacamole or through a Remote Desktop (RDP) client.  The gHost username and password are availble via [Guacamole](../tasks/Task-Connect-to-GNS3-VM-Guac.md).  A README.txt file on the gHost desktop contains your VM's hostname, the students credentials for the gHost, and other network resources that are assigned to that student.

Students involved in multiple ITS classes MAY be issued more than one gHost.  Each gHost has access to different networks, and as a result students MUST complete work for a class on the respective gHost, and MUST NOT use a gHost for other classwork.  

ITS Rule #21 - NEVER SHUTDOWN/POWER-OFF YOUR GHOST. You will need to contact the class faculty or ITS Lab Manager, Dr. Douglas Bowie (bowie@ohio.edu) to get the computer reenabled.

ITS Rule #35 - Unless given direct instructions from the Facult NEVER change the network configuration on the gHost.

ITS Rule #42 - Bring a towel and realize that gHosts can disapear.  Students should NEVER leave configurations or data uniquely on the gHost.  All computer operations are subject to failure.  Studnets are giving full administrative access to their gHost heightening the risks of failure. gHosts are given a disk Snapshot at the betting of the semester which will be used as a worst case fall back to recover any issues that are encountered during the semester.  All work on the gHost is lost if the snapshot is needed.  The ITS department faculty and staff are NOT responcible for the loss of work involving the need to recover a failed gHost.

## The Black Screen of Death

The current stack of services that provide the GUI Desktop and Remote Desktop in Ubuntu periodically suffer from a "Black Screen of Death".  Guacamole provides two SSH based interfaces to the gHost.  One runs the command that will correct the BlSOD, and other that provide direct terminal access that can be usehelpful in debugging connectivity problems

The following command fixes the Black Screen of Death, and is left for future reference. It should NOT normally be needed.

```DISPLAY=:10 xmessage -iconic x```
